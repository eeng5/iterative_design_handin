Part 0: Overview


The goal of this assignment was to perform an A/B test and statistical analysis to gather insight into two different UI designs. A/B testing is a user research technique, which is used to test whether the A or B version is a better design. This project involves simple A/B testing of two versions of a website. The metrics collected are then used to see which version among the baseline(A) or variant(B) performed better. The two images below depict the A and B versions of the interface for an online cactus shop website that can be found here. Note that either version A or version B will open randomly from this link, and that the changes are not complex since the focus of this project is the process of A/B testing, not interface design.

Part 1: Creating Hypotheses for Statistical Tests


The probability of a hypothesis given the data is very hard to compute; it is easier to compute the probability of the data given a hypothesis. Since we cannot hypothesize that we are correct from probable data, we try to disprove the opposite, or "null," hypothesis. We can do so using the following steps:

  1. State the null hypothesis and run an experiment.
  2. Find the probability the result would have occured given the null hypothesis.
  3. If that probability is less than 5%, then the alternate hypothesis is true.
I created null and alternative hypotheses for two metrics: Time to Completion (the time it takes to complete the desired task - in this case, order cacti) and Return Rate (the number of times a user returned to the page).


Time to Completion

Null Hypothesis: the time to completion for users of interface B is no better than the time to completion for users of interface A. In other words, the time to completion for users of interface B is either greater than that of A, or less than that of A but not to a statistically significant degree.

Alternative Hypothesis: the time to completion for users of interface B is better than the time to completion for users of interface A. Mathematically, this means that the time to completion for users of interface is B less than that of A by a statistically significant degree. In this case, B most likely performs better than A because the "Add to Cart" buttons are all evenly aligned, and therefore are easier to sequentially and efficiently select. Additionally, the user has to spend less time understanding the format of the page since the different items' sections are clearly delineated by horizontal lines, and the spacing allows for greater visual clarity.


Return Rate

Null Hypothesis: the return rate for users of interface B is no better than the return rate for users of interface A. In other words, the return rate for users of interface B is either greater than that of A, or less than that of A but not to a statistically significant degree.

Alternative Hypothesis: the return rate for users of interface B is better than the return rate for users of interface A. Mathematically, this means that the return rate for users of interface is B less than that of A by a statistically significant degree. In this case, B most likely performs better than A because B's layout is easier to understand, and therefore the user is immediately able to select the items they want and does not need to go back to review their choices before checking out. Additionally, since B's layout is more aesthetically-pleasing, the user will be motivated to make a purchase and will not be feel the need to second-guess themselves and return to review the original page before paying and exiting the site.

Part 2: Collecting Data and Applying Scripts


For this A/B test, I collected data from over 40 user sessions. The link provided in the Overview section, which randomly opens to interface A or B, was sent to my peers. They filled their carts with at least $150 of cacti. Each of their clicks was recorded, providing me with session-specific data including their date and time of access and their click targets.


The data was then extracted and filtered, and transferred to Excel for further processing. The processed data underwent statistical tests run by python scripts, the results of which are analyzed in Part 3.

Finally, here are my overall takeaways from this assignment about A/B testing in general:

  • Collecting data from a limited audience makes test results less meaningful. In this case, we were gathering data only from our fellow students who understood the goals of the data collection and tests, and this likely impacted how they interacted with the two interfaces.
  • It is possible that several small changes will not change the user experience in a way that will appear statistically significant, even if these changes dramatically improve a website's aesthetics (although aesthetics are subjective...).
  • A/B testing can inform creators about how users interact with their product. Importantly, creators may discover that changes that they consider important are not of significant value to users.
